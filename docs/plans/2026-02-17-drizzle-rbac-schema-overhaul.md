# PharmaPOS — Drizzle ORM + RBAC + Schema Overhaul Design

**Date:** 2026-02-17
**Status:** Approved
**Scope:** Full data layer migration to Drizzle ORM, schema correctness fixes, RBAC enforcement, business logic bug fixes, UX improvements
**Context:** No existing production databases. Testing phase only. Fresh schema only.

---

## 1. Motivation

The current raw-SQL data layer has produced three production SQLite errors during testing (`card_amount`, `updated_at`, `product_name`) caused by schema/code drift. The RBAC permission matrix is defined but never enforced. Eighteen schema issues were identified in a full audit. This design resolves all of them in a single pass.

---

## 2. Approach

**Drizzle ORM (Full Migration).** Replace `schema.sql` with a TypeScript Drizzle schema file as the single source of truth. Rewrite all service layer queries using Drizzle's query builder. Fix all schema issues, enforce RBAC at every IPC handler, fix all business logic bugs, and address two UX gaps (customer access at POS, AI vs Analytics nav duplication).

**Packages:**
- `drizzle-orm` — runtime query builder + type inference over better-sqlite3
- `drizzle-kit` (devDependency) — migration file generation from schema.ts

No new runtime processes. better-sqlite3 v12 stays as the driver.

## 4. Schema Design

### 4.1 Full Table Definitions

All tables are defined in `src/main/db/schema.ts` using Drizzle's `sqliteTable()` DSL. TypeScript types for every row are inferred via `typeof table.$inferSelect`.

#### `users`
```
id, username (unique), password_hash, full_name,
role CHECK('admin','manager','cashier'),
pin_code (bcrypt hash — not plain text),
is_active DEFAULT 1,
created_at, updated_at
```

#### `shifts`
```
id, user_id NOT NULL → users(id),
started_at DEFAULT now, ended_at,
opening_cash REAL NOT NULL DEFAULT 0,
closing_cash REAL, notes, status CHECK('active','closed')
```
Unique partial index: `(user_id) WHERE status = 'active'` — prevents two active shifts per user.
**Removed:** `expected_cash` column (computed at shift-end from actual sales, not maintained as running total).

#### `categories`
```
id, name (unique), description, parent_id → categories(id), created_at
```

#### `suppliers`
```
id, name, contact_person, phone, email, address,
lead_time_days DEFAULT 3, is_active DEFAULT 1, created_at
```

#### `products`
```
id, barcode (unique), sku (unique), name NOT NULL, generic_name,
category_id → categories(id), supplier_id → suppliers(id),
cost_price REAL DEFAULT 0, unit_price REAL NOT NULL,
tax_rate REAL DEFAULT 0, is_tax_inclusive INTEGER DEFAULT 1,
reorder_level DEFAULT 10, reorder_qty DEFAULT 50,
unit DEFAULT 'piece', is_active DEFAULT 1,
track_expiry DEFAULT 1, created_at, updated_at
```

#### `stock_batches`
```
id, product_id NOT NULL → products(id),
batch_number, quantity REAL NOT NULL DEFAULT 0 CHECK(quantity >= 0),
cost_price REAL NOT NULL, expiry_date, received_date DEFAULT today,
supplier_id → suppliers(id), created_at
```
**Changed:** `quantity` is `REAL` (was `INTEGER`) to support fractional units (liquid medications).

#### `sales`
```
id, receipt_number (unique, not null),
shift_id NOT NULL → shifts(id),
user_id NOT NULL → users(id),
customer_id → customers(id),
subtotal REAL NOT NULL DEFAULT 0,
tax_amount REAL NOT NULL DEFAULT 0,
discount_amount REAL NOT NULL DEFAULT 0,
discount_type CHECK('percentage','fixed'),
discount_value REAL DEFAULT 0,
total REAL NOT NULL DEFAULT 0,
payment_method NOT NULL CHECK('cash','card','mixed'),
cash_received REAL DEFAULT 0,
card_received REAL DEFAULT 0,
change_given REAL DEFAULT 0,
status DEFAULT 'completed' CHECK('completed','voided','refunded'),
void_reason,
customer_name, customer_phone,
created_at
```
**Changes:** `shift_id` is now `NOT NULL`. `customer_id` FK added directly (no migration hack).

#### `sale_items`
```
id, sale_id NOT NULL → sales(id) ON DELETE CASCADE,
product_id NOT NULL → products(id),
batch_id → stock_batches(id),
quantity INTEGER NOT NULL,
unit_price REAL NOT NULL,
cost_price REAL NOT NULL,
tax_rate REAL DEFAULT 0,
tax_amount REAL DEFAULT 0,
discount_amount REAL DEFAULT 0,
line_total REAL NOT NULL,
created_at
```
**Changes:** `tax_rate` and `tax_amount` now populated from product. `discount_amount` added for future per-item promotions.

#### `inventory_adjustments` ← NEW TABLE
```
id, product_id NOT NULL → products(id),
batch_id → stock_batches(id),
adjustment_type NOT NULL CHECK('add','remove','correction'),
quantity_change REAL NOT NULL,
reason TEXT,
user_id NOT NULL → users(id),
created_at
```

#### `sale_refunds` ← NEW TABLE
```
id, sale_id NOT NULL → sales(id),
user_id NOT NULL → users(id),
items TEXT NOT NULL (JSON array of {product_id, quantity, line_total}),
total_refunded REAL NOT NULL,
restock INTEGER DEFAULT 1,
reason TEXT,
created_at
```

#### `product_sales_daily`
```
PRIMARY KEY (date, product_id),   ← composite PK replaces UUID id
product_id NOT NULL → products(id),
quantity_sold INTEGER NOT NULL DEFAULT 0,
revenue REAL NOT NULL DEFAULT 0,
cost REAL NOT NULL DEFAULT 0,
profit REAL NOT NULL DEFAULT 0
```
**Changed:** UUID `id` removed; `(date, product_id)` becomes the primary key.

#### `customers`
```
id, name NOT NULL, phone UNIQUE,
email UNIQUE,
address, date_of_birth,
notes, is_active DEFAULT 1,
created_at, updated_at
```
**Changed:** `email` is now UNIQUE. `loyalty_points` removed — no loyalty system.

#### `audit_log`
```
id, timestamp DEFAULT now NOT NULL,
user_id → users(id), user_name,
action NOT NULL, entity_type NOT NULL,
entity_id, details TEXT (JSON), ip_address
```
Append-only. Never UPDATE or DELETE.

#### `settings`
```
key TEXT PRIMARY KEY, value NOT NULL, updated_at
```

## 5. Migration Strategy

No existing databases. Single migration: `0001_initial.sql`.

- Generated by `drizzle-kit generate` from `schema.ts`
- Applied at app startup via `migrate(db, { migrationsFolder })` from drizzle-orm
- Drizzle tracks applied migrations in `__drizzle_migrations` (auto-managed)
- WAL mode and `foreign_keys = ON` pragmas run before `migrate()`
- `drizzle-kit` is a devDependency only; runtime bundles the SQL files

```typescript
// src/main/db/index.ts
import Database from 'better-sqlite3'
import { drizzle } from 'drizzle-orm/better-sqlite3'
import { migrate } from 'drizzle-orm/better-sqlite3/migrator'
import * as schema from './schema'

export function initDatabase(dbPath: string) {
  const sqlite = new Database(dbPath)
  sqlite.pragma('journal_mode = WAL')
  sqlite.pragma('foreign_keys = ON')
  const db = drizzle(sqlite, { schema })
  migrate(db, { migrationsFolder: join(__dirname, 'migrations') })
  return db
}
```

## 6. Security Fixes

### 6.1 PIN Hashing

- `createUser` / `updateUser`: `pin_code = await bcrypt.hash(pin, 10)` if PIN provided
- `verifyPinCode`: `bcrypt.compare(submittedPin, storedHash)`
- No data migration needed (testing phase, no existing users)

### 6.2 RBAC Middleware

**`src/main/ipc/middleware.ts`**:
```typescript
export async function withPermission(
  userId: string,
  permission: Permission,
  handler: () => unknown
): Promise<unknown> {
  const user = getUserById(userId)
  if (!user || !hasPermission(user.role, permission)) {
    throw new Error(`Permission denied: ${permission}`)
  }
  return handler()
}
```

Every IPC handler that requires a permission wraps its logic in `withPermission(userId, permission, () => { ... })`. The `userId` is passed by the renderer with each call (already the pattern in `createSale`, shift handlers, etc.).

### 6.3 Permission Map

| Handler | Permission |
|---|---|
| `user:list/get` | `users:view` |
| `user:create` | `users:create` |
| `user:update` | `users:update` |

## 7. Business Logic Fixes

### 7.1 `expected_cash` void bug
- Remove `UPDATE shifts SET expected_cash = ...` from `createSale`
- Remove `expected_cash` column from schema
- Compute at shift-end: `SELECT COALESCE(SUM(cash_received),0) FROM sales WHERE shift_id=? AND status='completed'`

### 7.2 `sale_items` tax population
- `SaleItem` interface gains `tax_rate` (passed from product in POS store)
- Service calculates per-item `tax_amount`:
  - Tax-inclusive: `line_total - (line_total / (1 + tax_rate/100))`
  - Tax-exclusive: `line_total * (tax_rate / 100)`
- Both fields written to `sale_items`

### 7.3 Customer auto-linking on sale
- When `customer_phone` is provided in `createSale`, do a lookup:
  `SELECT id FROM customers WHERE phone = ?`
- If found, set `customer_id` on the sale record
- Text fields (`customer_name`, `customer_phone`) always written for receipt history

### 7.4 `inventory_adjustments` tracked properly
- Table added to schema (no more try-catch swallow)
- `adjustStockBatch` service writes to this table on every adjustment
- Feeds existing `STOCK_ADJUSTED` audit log entry

### 7.5 `sale_refunds` service
- `createRefund(saleId, userId, items, reason, restock)`:
  - Validates sale exists and is `'completed'`
  - Inserts `sale_refunds` row
  - Updates `sales.status = 'refunded'`
  - If `restock = true`: increments batch quantities for each returned item
  - Logs `SALE_REFUNDED` to audit log
  - Requires `sales:refund` permission

### 7.6 `voidSale` service (new)
- Validates `void_reason` is non-empty string
- Updates `sales.status = 'voided'`, sets `void_reason`
- Logs `SALE_VOIDED` to audit log
- Requires `sales:void` permission

### 7.7 `updated_at` maintained correctly
- Every Drizzle `update()` call includes `.set({ updatedAt: new Date().toISOString() })`
- Shared `now()` helper: `() => new Date().toISOString()`

### 7.8 `stock_batches.quantity` → REAL
- Schema type: `REAL` (was `INTEGER`)
- Supports fractional quantities (liquid medications sold in mL)
- FEFO logic already handles decimals via `Math.min()`

### 7.9 Receipt number — confirmed safe
- Runs inside `db.transaction()` with synchronous better-sqlite3
- No concurrency gap. No change needed.

## 8. Frontend Changes

### 8.1 RBAC Hook + Gate

```typescript
// src/renderer/src/hooks/usePermission.ts
export function usePermission(permission: Permission): boolean {
  const role = useAuthStore(s => s.user?.role)
  return role ? hasPermission(role, permission) : false
}

// src/renderer/src/components/PermissionGate.tsx
export function PermissionGate({ permission, fallback, children }) {
  const allowed = usePermission(permission)
  return allowed ? children : (fallback ?? null)
}
```

### 8.2 Sidebar Role Filtering

| Nav item | Required permission | Visible to |
|---|---|---|
| POS | `sales:create` | All |
| Inventory | `inventory:view` | All |
| Analytics | `reports:view` | Manager, Admin |
| Customers | `sales:view_all` or `sales:create` | All (cashier: read-only) |
| Users | `users:view` | Manager, Admin |
| Audit | `reports:view` | Manager, Admin |
| Settings | `settings:view` | Admin |

**AI Insights nav item removed** (was a duplicate of Analytics). AI Insights content moves to a dedicated tab inside `AnalyticsPage`.

### 8.3 Customer Lookup at POS

In `PaymentModal`, when the cashier types a phone number into the Customer Phone field:
- Debounced lookup: `window.electron.customers.getByPhone(phone)`
- If found: auto-fills `customer_name` and shows the customer's **purchase history panel** inline — a scrollable list of their past transactions showing date, items bought (name + quantity), and total
- The cashier can use this to help a returning patient reorder: "Last time you bought Paracetamol 500mg x2 and Amoxicillin 250mg x1" — the cashier adds those items to the cart manually
- If not found: remains a walk-in capture (name/phone stored on sale but no `customer_id`)
- History panel is read-only — no navigation away from POS required

### 8.4 AI Insights Tab in AnalyticsPage

`AnalyticsPage` gains a new "AI Insights" tab alongside existing report tabs. The tab renders:
- Reorder suggestions (calls existing `ai:getReorderSuggestions`)
- Sales forecast (calls existing `ai:getSalesForecast`)
- Dead stock detection (calls existing `ai:getDeadStockDetection`)

The `'ai'` case in `MainLayout` is removed. The `'ai'` nav item is removed from `Sidebar`.

### 8.5 Page-level Buttons Gated

Inside pages, action buttons (Delete Product, Void Sale, Create User, etc.) use `<PermissionGate>` — disabled with tooltip if user lacks permission.
